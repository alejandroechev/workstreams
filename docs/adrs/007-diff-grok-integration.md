# ADR 007: diff-grok review skill + Diff Review tile

## Status
Accepted.

## Date
2026-05-26

## Context

Reviewing code generated by Copilot CLI agents is a routine part of how this
project is built (the discipline harness in ADR 002 already enforces tests +
visual + docs sub-todos for every feature). The existing user-level skill
`code-grok` is great for understanding code, but it has two gaps when used as
a review tool:

1. It is not exhaustive. It picks interesting questions but does not walk
   through every change in a diff, which is what a real code review demands.

2. It has no place to capture review comments. After a review session, the
   user has to mentally hold the findings and then dispatch a separate agent
   to apply them — there is no structured artifact bridging the two phases.

A diff-focused sibling skill is the natural shape. It must also integrate
with Workstreams (the surrounding desktop app) so reviews persist across
crashes and are tied to the workstream the user is driving.

## Decision

1. **Introduce a sibling skill `diff-grok`** at `~/.copilot/skills/diff-grok/`.
   It is exhaustive — every non-trivial hunk is acknowledged — and captures
   per-hunk comments. Triviality is judged by the agent; whitespace,
   imports, renames, generated files, and version bumps are typically
   trivial.

2. **Add a custom `diff_review` tile type** to Workstreams. The tile is the
   primary interaction surface per "chunk" (semantic cluster of hunks,
   cross-file allowed). It renders the diff with Monaco's diff editor, the
   agent's current question, and an inline comment input pinned to a
   selected line range.

3. **Skill drives, tile interacts.** The skill (running in an adjacent
   terminal-style tile in the same workstream) posts a chunk into the
   Workstreams SQLite DB and waits. The tile renders the chunk, accepts
   comments + an ack from the user, and writes the result back. The skill
   wakes and advances to the next chunk.

4. **Two-layer communication.** Tauri events (`diff-review:chunk-active`,
   `diff-review:chunk-done`) for instant nudges; SQLite is the source of
   truth so a crashed terminal or webview can resume exactly where it left
   off. Event names are sanitized identically on both sides (see ADR 006's
   note on Tauri event-name constraints).

5. **Diff sources are pluggable, chosen at session start.** v1 supports:
   - branch vs base (`git diff <base>...HEAD`)
   - GitHub PR (`gh pr diff <number>`)
   - working tree (staged + unstaged)

6. **One active review per workstream.** Concurrent reviews per workstream
   are out of scope for v1.

7. **Persistence model.** Active reviews live in the workstream's SQLite DB.
   On completion, the skill exports a rich hierarchical JSON document plus
   a human-readable markdown "action plan" to
   `.copilot-reviews/<review-id>/` inside the repo. The DB row flips to
   `archived` but remains queryable.

8. **Comment model is minimal.** A comment carries `text`, a code anchor
   (file + line range), and is associated with a chunk. The chunk itself
   carries an `approval_state` of `pending`, `seen`, `approved`, or
   `commented`. There is no per-comment priority, intent tag, or
   threading in v1 — the markdown action plan is the bridge to the
   actioning agent.

9. **Bundling is semantic.** The skill does a pre-analysis pass over the
   full diff at session start and clusters hunks into chunks by semantic
   theme. Cross-file bundles are allowed.

10. **Drift handling.** Each chunk activation re-verifies hunk hashes
    against the current diff. If drift is detected, affected chunks are
    marked `invalidated`, the user is warned in the tile, and re-ack is
    required after re-fetch.

11. **Completion is strict.** Review reaches `completed` only when every
    non-trivial hunk is acknowledged (approved or commented). Trivial
    hunks are auto-acked at chunk-activation time.

12. **Out of scope for v1:** GitHub PR comment posting, screenshots in
    comments, review templates (security focus, perf focus presets),
    multi-reviewer collaboration, comment threading, in-diff search /
    filter, action-agent auto-dispatch, overlays (blame, related tests).
    The action handoff is a manual user step: open the markdown action
    plan, paste into a new Copilot CLI session.

## Schema

```sql
CREATE TABLE diff_reviews (
    id TEXT PRIMARY KEY,
    workstream_id TEXT NOT NULL REFERENCES workstreams(id) ON DELETE CASCADE,
    diff_source TEXT NOT NULL,        -- 'branch' | 'pr' | 'working_tree'
    source_ref TEXT,                  -- base ref, PR number, or NULL for working tree
    status TEXT NOT NULL DEFAULT 'planning',  -- planning|active|completed|archived
    plan_json TEXT,                   -- raw cluster plan from pre-analysis
    exported_path TEXT,               -- path of JSON+md export when completed
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL,
    completed_at TEXT
);

CREATE TABLE diff_chunks (
    id TEXT PRIMARY KEY,
    review_id TEXT NOT NULL REFERENCES diff_reviews(id) ON DELETE CASCADE,
    ordinal INTEGER NOT NULL,
    title TEXT NOT NULL,
    summary TEXT,
    is_trivial INTEGER NOT NULL DEFAULT 0,
    state TEXT NOT NULL DEFAULT 'pending',  -- pending|seen|approved|commented|invalidated
    question_text TEXT,
    question_style TEXT,              -- 'socratic'|'guided'|'review'
    invalidated_at TEXT,
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL
);

CREATE TABLE diff_hunks (
    id TEXT PRIMARY KEY,
    chunk_id TEXT NOT NULL REFERENCES diff_chunks(id) ON DELETE CASCADE,
    file_path TEXT NOT NULL,
    old_start INTEGER,
    old_lines INTEGER,
    new_start INTEGER,
    new_lines INTEGER,
    patch_text TEXT NOT NULL,
    content_hash TEXT NOT NULL        -- sha256 of patch_text for drift detection
);

CREATE TABLE diff_comments (
    id TEXT PRIMARY KEY,
    chunk_id TEXT NOT NULL REFERENCES diff_chunks(id) ON DELETE CASCADE,
    anchor_file TEXT NOT NULL,
    anchor_line_start INTEGER NOT NULL,
    anchor_line_end INTEGER NOT NULL,
    text TEXT NOT NULL,
    created_at TEXT NOT NULL
);

CREATE TABLE diff_review_events (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    review_id TEXT NOT NULL REFERENCES diff_reviews(id) ON DELETE CASCADE,
    event_type TEXT NOT NULL,
    payload_json TEXT NOT NULL,
    created_at TEXT NOT NULL
);

CREATE INDEX idx_diff_chunks_review ON diff_chunks(review_id, ordinal);
CREATE INDEX idx_diff_hunks_chunk ON diff_hunks(chunk_id);
CREATE INDEX idx_diff_comments_chunk ON diff_comments(chunk_id);
CREATE INDEX idx_diff_review_events_review ON diff_review_events(review_id, id);
```

## Event contracts

Both sides agree on these names (sanitization-safe under Tauri's rules):

| Event                          | Direction        | Payload                                |
|--------------------------------|------------------|----------------------------------------|
| `diff-review:plan-ready`       | backend → tile   | `{ reviewId, chunkCount }`             |
| `diff-review:chunk-active`     | backend → tile   | `{ reviewId, chunkId, ordinal }`       |
| `diff-review:chunk-done`       | tile → backend   | `{ reviewId, chunkId, state }`         |
| `diff-review:comment-added`    | tile → backend   | `{ reviewId, chunkId, commentId }`     |
| `diff-review:drift-detected`   | backend → tile   | `{ reviewId, chunkIds }`               |
| `diff-review:completed`        | backend → tile   | `{ reviewId, exportedPath }`           |

The skill subscribes to `diff-review:chunk-done` to know when to advance.

`.copilot-reviews/<review-id>/review.json`:

```json
{

  "review_id": "rev-2026-05-26-abc",
  "workstream_id": "ws-...",
  "diff_source": "branch",
  "source_ref": "master",
  "completed_at": "2026-05-26T15:00:00Z",
  "chunks": [
    {

      "hunks": [
        { "file": "src/auth/mw.ts", "old_start": 42, "new_start": 42, "patch": "@@ ..." }
```

Plus `action-plan.md` — the same content rendered as a checklist for a
follow-on Copilot CLI session.

## Consequences

- **Reusable comment store.** Reviews accumulate as historical artifacts in
  the repo, queryable by other tools later.
- **Monaco reuse.** We already ship Monaco for file editing (ADR 006); the
  diff editor reuses the same no-op worker bootstrap, so no new bundle
  weight.
- **No GitHub coupling in v1.** Reviews work entirely against local git
  state. PR-posting is a clear v2 follow-up once the local loop is solid.
- **One-active-per-workstream limit.** Simplifies the UI; if you need to
  compare two PRs, create two workstreams.
- **Skill ↔ tile cooperation pattern.** This is the first feature where a
  user-level skill cooperates with a custom Workstreams tile. The pattern
  (event nudges + SQLite source of truth) is generalizable to future
  features.

## Alternatives considered

- **Use the existing repo-explorer tile.** Rejected: file-tree-first UX
  doesn't match a review-of-changes workflow, and we'd have to bolt comments
  into a tile not designed for them.
- **Build it as a standalone Electron/Web app.** Rejected: defeats the point
  of Workstreams as the unified surface, and we'd have to reinvent session
  persistence.
- **Reuse `code-grok` and add a `--exhaustive --diff` flag.** Rejected: the
  comments workflow and tile integration are large enough additions that a
  separate skill keeps `code-grok` focused on its current strength
  (comprehension).

## Tile entry points (added post-launch)

Added: 2026-05-26 after gap-analysis discussion.

Diff Review tiles can be opened through two supported paths:

1. **Skill-driven auto-open.** After `diff-grok` submits its plan with
   `set_review_plan`, it calls `create_or_focus_diff_review_tile(workstream_id,
   review_id)`. This makes the planned review visible without requiring the user
   to manually add a tile.

2. **User-driven open.** The Add Tile menu exposes Diff Review via `Alt+G`. That
   path lists active reviews for the workstream and opens the selected review
   using the same backend command.

The tile-opening command is idempotent: the same `(workstream_id, review_id)`
returns the same `diff_review` tile instead of creating duplicates. Both entry
points therefore share the same resume-safe contract.

When a new tile is inserted, Rust emits `tile-created` with the full `Tile`
struct as its payload. The React listener routes the update by
`tile.workstream_id`, not by the currently active workstream. If that workstream
is not loaded in frontend state, the listener is a no-op; first-load still reads
truth from SQLite, so no event queue is required.
